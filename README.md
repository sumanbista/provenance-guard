# Provenance Guard

A backend service that a creative-sharing platform can plug in to answer one question about
submitted text: **was this written by a human, or generated by AI?** It returns an
attribution verdict, a calibrated confidence score, and a plain-language transparency label —
and (from Milestone 5) lets creators appeal a classification.

Provenance Guard is **not** a content police. Its goal is honest attribution and trust, which
means *acknowledging uncertainty* rather than forcing a binary answer. A guiding principle
runs through every design decision:

> **The false-positive asymmetry.** On a writing platform, wrongly branding a *human's* work
> as AI-generated is the damaging error — worse than failing to catch a piece of AI text. So
> the system requires *stronger* evidence to say "AI" than to say "human," and when its two
> signals disagree it defaults to *Uncertain* rather than making an accusation.

Full architecture, flow diagrams, and design rationale live in [planning.md](planning.md).

|--------|----------|---------|
| `POST` | `/submit` | Submit text for attribution. Body: `{ "text": str, "creator_id"?: str }`. Rate-limited (see below). Returns attribution + confidence + transparency label + both signal scores. |

## Detection Pipeline (Multi-Signal)

Attribution uses **two independent signals**. They are genuinely *distinct* because they
measure different properties and **fail in different ways**: one is semantic and model-based,
the other structural and rule-based. Agreement between them is real evidence; disagreement is
the cue to report *Uncertain*.

| | Signal 1 — LLM-as-judge | Signal 2 — Stylometry |
|---|---|---|
| **Type** | Semantic (model-based) | Structural (pure Python) |
| **Measures** | Whether the text *reads* as human vs. machine — coherence, phrasing, voice, idiosyncrasy | Statistical *shape* of the text — sentence-length variability and punctuation variety |
| **Why it separates** | AI prose is smooth, generic, evenly balanced; human writing has quirks and uneven emphasis | Humans are "bursty" (varied sentence lengths); AI is uniform |
| **Output** | `llm_score ∈ [0,1]` + one-line reason | `sty_score ∈ [0,1]` + raw metrics |
| **Blind spot** | Non-deterministic; fooled by edited/"humanized" AI and by formal/generic human text | Genre-blind (formal essays look uniform); needs length to be stable; ignores meaning |

### Signal 1 — LLM-as-judge (Groq `llama-3.3-70b-versatile`)

The model is prompted to return a fixed, parseable format (`AI_LIKELIHOOD: 0–100` + `REASON`),
which we normalize to `llm_score ∈ [0,1]`. It **fails closed**: any API error or unparseable
output returns a neutral `0.5`, so a transient failure can never push a submission toward an
AI accusation. Implemented in [detection/llm_signal.py](detection/llm_signal.py).

### Signal 2 — Stylometry (pure Python)

Two metrics, combined with a weighted average into `sty_score` (higher = more AI-like):

1. **Sentence-length burstiness** (primary, weight 0.75) — the coefficient of variation
   (`std/mean`) of sentence lengths. Humans vary a lot (high CV); AI is uniform (low CV).
2. **Punctuation variety** (secondary, weight 0.25) — how many distinct punctuation marks are
   used. Humans use more; AI is plainer.

Implemented in [detection/stylometry.py](detection/stylometry.py); calibration constants live
in [config.py](config.py). (A type-token-ratio metric was tested and **dropped** — see
[Calibration & Tuning](#calibration--tuning-milestone-4) below.)

## Confidence Scoring & Uncertainty

The two signals are combined into a single verdict and a confidence score. Implemented in
[detection/scoring.py](detection/scoring.py).

**What confidence means (decided before the math):** `confidence ∈ [0,1]` is the system's
confidence in the verdict it reports. **`0.5` means "we genuinely cannot tell"** — the honest
center of the scale. As it approaches `1.0`, we are near-certain in the stated direction.

```
p_ai       = 0.6 * llm_score + 0.4 * sty_score     # weighted blend (LLM more reliable)
agreement  = 1 - |llm_score - sty_score|           # 1 = signals fully agree
base_conf  = |p_ai - 0.5| * 2                       # 0 at the midpoint, 1 at the extremes
confidence = base_conf * (0.5 + 0.5 * agreement)    # signal disagreement halves confidence
direction  = "AI" if p_ai >= 0.5 else "human"
```

Multiplying by `(0.5 + 0.5·agreement)` means two signals that contradict each other can never
produce high confidence — the mechanism that turns disagreement into *Uncertain*.

**Asymmetric verdict thresholds** (the false-positive protection), evaluated in order:

| Condition | Verdict |
|-----------|---------|
| `word_count < 60` (too short to judge reliably) | **uncertain** (confidence capped at 0.30) |
| `direction == "AI"` **and** `confidence ≥ 0.75` **and** `agreement ≥ 0.5` | **likely-ai** |
| `direction == "human"` **and** `confidence ≥ 0.60` | **likely-human** |
| otherwise | **uncertain** |

Note the asymmetry: declaring **AI** needs a *higher* confidence (0.75) **and** signal
agreement; declaring **human** needs only 0.60. A high `p_ai` produced by one signal alone
therefore falls through to *uncertain* instead of accusing a human. A `0.51` confidence always
lands in *uncertain*; a `0.95` clears a bar — the score meaningfully changes the outcome.

## Transparency Label

Every `/submit` response includes a plain-language `label` written for a **non-technical
reader**. There are exactly **three variants**, selected by the verdict, and each embeds the
live confidence as a percentage (`our certainty: N%`) so a 0.51 and a 0.95 result read
differently. Implemented in [labels.py](labels.py).

The wording reflects the false-positive asymmetry: the **AI** label is the most hedged and is
the only one that proactively promises human review, because wrongly accusing a human is the
costly error.

**The exact text of all three variants** (`N` is the confidence rounded to a whole percent):

**1. High-confidence AI** (verdict `likely-ai`):

> 🤖 Likely AI-generated. Multiple independent signals suggest this text was probably produced by an AI system (our certainty: N%). Automated detection is not perfect and can misclassify original human writing. If you wrote this yourself, you can appeal this result and a person will review it.

**2. High-confidence human** (verdict `likely-human`):

> ✍️ Likely written by a human. Our analysis found the writing patterns and style consistent with human authorship (our certainty: N%). This is an automated assessment, not a guarantee. If you believe it's wrong, you can appeal.

**3. Uncertain** (verdict `uncertain` — signals disagree, low confidence, or text too short):

> ❓ Attribution uncertain. Our signals disagree or aren't strong enough to reliably determine whether this text was written by a human or generated by AI (our certainty: N%). We are not making a determination — please treat this as inconclusive rather than as evidence either way.

### Example `/submit` response

```json
{
  "content_id": "9dc666e9-7b4c-4cbc-a328-1b1f52487377",
  "creator_id": "writer-anna",
  "attribution": "likely-ai",
  "confidence": 0.8066,
  "label": "🤖 Likely AI-generated. Multiple independent signals suggest this text was probably produced by an AI system (our certainty: 81%). Automated detection is not perfect and can misclassify original human writing. If you wrote this yourself, you can appeal this result and a person will review it.",
  "signals": {
    "llm": { "llm_score": 0.9, "reason": "..." },
    "stylometry": { "sty_score": 0.9167, "metrics": { "sentence_len_cv": 0.13, "distinct_punctuation": 1, "ttr": 0.77 } }
  },
  "scoring": { "p_ai": 0.907, "agreement": 0.983, "direction": "AI", "too_short": false },
  "status": "classified"
}
```

## Calibration & Tuning (Milestone 4)

The stylometry metrics were not left at their initial guessed thresholds — they were **tuned
against a labeled calibration set** of 11 samples (5 human, 6 AI), spanning casual and formal
writing. The measured metric ranges overturned two of the three original metrics:

| Metric | HUMAN (min/mean/max) | AI (min/mean/max) | Verdict |
|--------|----------------------|-------------------|---------|
| **Burstiness (CV)** | 0.32 / 0.51 / 0.62 | 0.13 / 0.25 / 0.38 | ✅ **Clean separator** — kept as primary |
| **Punctuation variety** | 1 / 2.6 / 5 | 1 / 1.67 / 3 | ⚠️ Weak but real — kept as light secondary |
| **Type-token ratio** | 0.62 / 0.71 / 0.87 | 0.70 / 0.79 / 0.90 | ❌ **Dropped** — see below |

**Why TTR was dropped:** its ranges overlap almost entirely, and its direction was
**reversed** on real data — AI text had *higher* lexical diversity (0.79) than human text
(0.71), because longer human essays naturally repeat words. Our original "low diversity = AI"
rule would have **flagged long human essays as AI** — the exact false positive we must avoid.
TTR is still computed and reported for transparency, but excluded from the score.

**Changes made, all data-driven:**
- Dropped TTR from the score; **weighted burstiness 0.75 / punctuation 0.25**.
- Recalibrated the burstiness band to the observed human/AI boundary (`CV 0.25 → 0.50`).
- Raised `MIN_WORDS` from 40 → **60**: a length-sensitivity probe showed burstiness is noisy
  below ~60 words, so shorter text now honestly returns *uncertain*.

Effect on stylometry scores (samples clearing the 60-word gate):

| Sample | words | sty before | sty after |
|--------|-------|-----------|-----------|
| AI Sample-1 | 111 | 0.29 | **0.75** |
| AI Sample-2 | 109 | 0.56 | **0.92** |
| AI Sample-4 | 173 | 0.56 | **0.92** |
| Human-1 | 129 | 0.22 | **0.17** |
| Human-2 | 207 | 0.00 | **0.00** |

Human/AI separation improved from AI-mean 0.42 vs human-mean 0.17 (barely distinguishable) to
**AI-mean 0.73 vs human-mean 0.24**.

### Why we tuned — system behavior before vs. after

A miscalibrated scoring function undermines everything built on top of it, so this was not
optional polish. With the initial guessed thresholds the pipeline had two concrete problems:

- **The `likely-ai` verdict was effectively unreachable for real input.** Stylometry averaged
  one good metric (burstiness) with one dead one (punctuation rarely fired) and one *harmful*
  one (TTR), so genuine AI paragraphs scored only ~0.29–0.56. Combined with the LLM, even
  clearly-AI text produced low confidence and fell to `uncertain`. A detector that can *never*
  say "AI" isn't useful — and "uncertain" on obvious AI is misleading, not honest.
- **A latent false-positive risk.** TTR's reversed direction meant long human essays (lower
  lexical diversity) were being pushed *toward* AI — the exact error the whole system is
  designed to avoid.

We tuned to get three properties at once: (1) the AI label reachable for genuine AI, (2) no
false positives on real human writing, and (3) honest `uncertain` for genuinely ambiguous or
too-short text. **After tuning:** real long AI reaches `likely-ai` at confidence ~0.81,
humans are never falsely flagged, ambiguous/short text returns `uncertain`, and the full
calibration set shows **11/11 no dangerous misclassifications** (see next section).

*(Reproduce with `python scripts/calibrate_stylometry.py` and
`python scripts/length_sensitivity.py`.)*

## Testing the Scoring

The scoring function was validated on **deliberately chosen inputs** — text confidently
AI-generated, text confidently human-written, and borderline cases — to check that the scores
match intuition. The full pipeline (both signals + scoring) was run end-to-end on the 11-sample
calibration set (`python scripts/test_pipeline.py`):

```

----------------------------------------------------------------------------
HUMAN   0.20  0.08  0.153  0.883  0.300  uncertain      Clearly human (short, <60 words)
AI      0.90  0.53  0.751  0.629  0.300  uncertain      Clearly AI (short, <60 words)
AI      0.60  0.35  0.502  0.754  0.003  uncertain      Borderline: lightly edited AI
AI      0.90  0.75  0.840  0.850  0.629  uncertain      AI Sample-1
AI      0.90  0.92  0.907  0.983  0.807  likely-ai      AI Sample-2
AI      0.80  0.92  0.847  0.883  0.653  uncertain      AI Sample-3
AI      0.90  0.92  0.907  0.983  0.807  likely-ai      AI Sample-4
HUMAN   0.20  0.17  0.187  0.967  0.616  likely-human   Human Sample-1
HUMAN   0.20  0.00  0.120  0.800  0.684  likely-human   Human Sample-2
HUMAN   0.20  0.63  0.371  0.573  0.203  uncertain      Human Sample-3 (formal essay)
HUMAN   0.20  0.34  0.257  0.859  0.453  uncertain      Human Sample-4
----------------------------------------------------------------------------
no dangerous misclassifications: 11/11
```

**Do the scores match intuition? Yes — and the failures fail safe:**

- **Confident human** (Human-1, Human-2) → `likely-human` at high confidence. ✅
- **Confident AI** (AI Sample-2, AI Sample-4) → `likely-ai` at confidence 0.81. ✅
- **Borderline / hard cases** → `uncertain`, never a wrong accusation:
  - *Human-3* is a formal essay with unusually uniform sentences (stylometry 0.63, AI-leaning),
    but the LLM reads it as human — the signals disagree, confidence drops to 0.20, and the
    verdict is `uncertain`, **not** a false AI accusation. This is the asymmetry protecting a
    real human writer.
  - *Short texts* (<60 words) → `uncertain` with capped confidence, because the structural
    signal isn't reliable on so little text.
  - *AI Sample-1 / AI Sample-3* are genuinely AI, but confidence lands just below the strict
    0.75 AI bar → `uncertain`. The system prefers "unsure" over a possibly-wrong "AI."

**Zero dangerous misclassifications** across all 11 samples — no human was labeled AI, and no
AI was labeled human. The system is intentionally conservative about the `likely-ai` verdict,
which is the correct bias for a writing platform.

## Appeals Workflow

Automated attribution is imperfect, so creators can contest a classification via
`POST /appeal`. No automated re-classification happens — an appeal routes the case to a human.

- **Who can appeal:** the creator, using the `content_id` returned at submission.
- **What they provide:** `{ "content_id": str, "creator_reasoning": str }` — the id plus a
  free-text explanation of why they believe the verdict is wrong.
- **What the system does** ([app.py](app.py) `/appeal`, [db.py](db.py) `record_appeal`):
  1. Looks up the submission by `content_id` (returns **404** if it doesn't exist).
  2. Records the appeal in the `appeals` table (`appeal_id`, `content_id`, `creator_reasoning`, timestamp).
  3. Flips the submission's status `classified` → **`under_review`**.
  4. Logs an `event: "appeal"` row in the audit log **alongside the original decision** — the
     original `attribution`, `confidence`, and both signal scores are copied in next to the
     `creator_reasoning`, so a reviewer sees the full contested context in one row.
- **Response:** `{ content_id, appeal_id, status: "under_review", message }`.
- **What a human reviewer sees:** via `GET /log`, the appeal entry carries the original verdict,
  its confidence, both signal scores, and the creator's stated reason — everything needed to
  re-adjudicate by hand.

**Example:**

```bash
curl -s -X POST http://localhost:5000/appeal -H "Content-Type: application/json" \
  -d '{"content_id": "9dc666e9-7b4c-4cbc-a328-1b1f52487377",
       "creator_reasoning": "I wrote this analysis myself for a university economics course."}'
```
```json
{
  "content_id": "9dc666e9-7b4c-4cbc-a328-1b1f52487377",
  "appeal_id": "…",
  "status": "under_review",
  "message": "Appeal received. Your submission is now under review by a human."
}
```

## Rate Limiting

`POST /submit` is rate-limited with [Flask-Limiter](https://flask-limiter.readthedocs.io/),
keyed by client IP, using in-memory storage. Exceeding a limit returns **HTTP 429** with a JSON
body (`{ "error": ..., "limit": ... }`).

**Chosen limits: `5 per minute; 30 per hour; 100 per day`.**

These are tuned to how a creative writer *actually* uses the service. Submitting a finished
piece is inherently low-frequency: writing and editing take minutes to hours, so submissions
are naturally spaced out. The realistic burst is a writer re-checking after edits (submit →
read → tweak → resubmit), a cycle of tens of seconds. And because **every `/submit` triggers
one Groq LLM call**, the caps also bound cost and latency, not just request count.

| Limit | Protects against | Why this number |
|-------|------------------|-----------------|
| **5 / minute** | A script *flooding* the endpoint | One submission every 12s comfortably covers a writer resubmitting after edits; a flood script trips it within seconds. It's also near the practical ceiling anyway, since each call waits on a ~1–3s LLM round-trip. |
| **30 / hour** | *Paced* abuse under the minute cap | A heavy-but-legitimate drafting session (clearing a backlog of pieces) fits; an adversary pacing requests *just under* 5/min to stay quiet is still stopped. |
| **100 / day** | Cost blow-up and slow-drip probing | No single writer legitimately analyzes 100 distinct pieces in a day. Bounds total daily Groq spend and long-horizon probing of the detector. |

The three tiers are layered defense: the per-minute cap stops *fast* abuse, the hour/day caps
stop *slow* abuse. The numbers are generous for a genuine writer but far below what automated
abuse requires.

### Verification (evidence)

Sending 12 rapid `POST /submit` requests from one client (limit: 5/min) — the first 5 succeed,
the rest are throttled:


```bash
  curl -s -o /dev/null -w "%{http_code}\n" -X POST http://localhost:5000/submit \
```

**Deployment caveat:** limits are keyed by IP, which is right for this project. A real platform
integrating Provenance Guard would instead key by authenticated API client or `creator_id` —
otherwise all traffic arriving from the platform's own servers would share one IP and one
bucket. The limit values live in [config.py](config.py) (`RATE_LIMIT_SUBMIT`) for easy tuning.

## Audit Log

Every attribution decision **and** every appeal is written as a structured row to the SQLite
`audit_log` table ([db.py](db.py)) and served as JSON via `GET /log`. It is a structured store,
not console output. Each row captures:

| Column | Meaning |
|--------|---------|
| `timestamp` | ISO-8601 UTC (`Z`-suffixed) |
| `content_id` | the submission's unique id |
| `creator_id` | who submitted it (optional) |
| `event` | `submission` or `appeal` |
| `attribution` | `likely-ai` / `likely-human` / `uncertain` |
| `confidence` | combined confidence score |
| `llm_score` | signal 1 (LLM) individual score |
| `sty_score` | signal 2 (stylometry) individual score |
| `status` | `classified` or `under_review` |
| `appeal_reasoning` | the creator's reason (only on `appeal` events) |

Because an `appeal` event copies the original decision's scores alongside the reasoning, whether
a piece has been appealed — and the full context of the contested decision — is visible directly
in the log.
